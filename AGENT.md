# Let's Peppol Project Documentation

Let's Peppol is an open-source platform designed to simplify e-invoicing and document exchange over the Peppol network. It provides a user-friendly interface for managing invoices, partners, and products, while handling the complexities of UBL (Universal Business Language) validation and Peppol network communication behind the scenes.

## 🏗️ Architecture

The project follows a microservices-inspired architecture, containerized with Docker, comprising the following components:

- **Frontend (`app/ui`)**: Built with **Aurelia 2**, **Vite**, and **TypeScript**. It provides a responsive, localized (EN, FR, NL, DE) user interface.
- **Application Backend (`app/backend`)**: A **Spring Boot (Java 21)** service responsible for core business logic, including:
  - Document management and UBL processing.
  - Partner and product management.
  - PDF generation via XSLT-based rendering.
  - Integration with Open Collective for billing/donations.
- **KYC Service (`kyc`)**: A **Spring Boot** service focused on:
  - User registration and authentication.
  - Identity verification using eID.
  - Integration with the Belgian Crossroads Bank for Enterprises (KBO) to verify company data.
- **Peppol Proxy (`proxy`)**: A **Spring Boot** service acting as the gateway to the Peppol network:
  - Communicates with External Access Point APIs (Scrada, E-Invoice).
  - Manages document synchronization, sending, and receiving.
  - Handles registry lookups and Peppol ID registrations.
- **Database**: A shared **PostgreSQL 16** instance with separate users and schemas for each service to ensure data isolation.

## 🔄 Code Flow

1. **User Interaction**: Users interact with the Aurelia frontend. Requests requiring authentication include a JWT token in the `Authorization` header.
2. **Identity & Auth**: The `kyc` service handles initial registration and login. Once authenticated, tokens are used to authorize requests to the `app/backend`.
3. **Business Operations**: The `app/backend` processes business logic (e.g., creating an invoice). It validates UBL XML using the **PHIVE** engine and stores metadata in the database.
4. **Peppol Exchange**: When a user sends an invoice, `app/backend` communicates with the `proxy` service. The `proxy` then forwards the document to the Peppol network through specialized external APIs.
5. **Background Sync**: The `proxy` service contains schedulers that periodically poll for incoming documents and status updates from the Peppol network, which are then synced back to the main application.

## 🚀 Feature Flow

### 1. Onboarding & KYC

1. **Search**: User enters a VAT number; `kyc` fetches company details from the KBO registry.
2. **Verification**: After email confirmation, the user undergoes identity verification (eID).
3. **Activation**: Upon successful KYC, the Peppol identity is activated through the `proxy` service.

### 2. Document Management

1. **Creation**: Invoices can be created manually or uploaded as UBL XML.
2. **Validation**: The system uses the PHIVE engine to ensure compliance with EN16931 and Peppol BIS Billing 3.0 rules.
3. **PDF Preview**: Documents can be rendered as PDFs for human review before sending.
4. **Sending**: Validated documents are sent via the `proxy` to the recipient's Peppol ID.

### 3. Partner & Product Management

- Users can maintain a directory of partners (customers/suppliers) and products/services to speed up invoice creation.

## 🔌 API Guide

### KYC Service (`:8081` internally)

- `GET /api/register/company/{peppolId}`: Fetch company info for registration.
- `POST /api/register/confirm-company`: Start email activation.
- `POST /api/auth/login`: Authenticate and receive JWT.
- `POST /api/identity-verification/prepare`: Initiate eID verification.

### Application Backend (`:8080` internally)

- `GET /sapi/document`: List and filter invoices.
- `POST /sapi/document`: Create or upload an invoice.
- `PUT /sapi/document/{id}/send`: Dispatch invoice to Peppol.
- `GET /sapi/document/{id}/pdf`: Generate PDF rendering.
- `GET /sapi/partner`: Manage partner directory.

### Proxy Service (`:8082` internally)

- `GET /api/registry/lookup/{peppolId}`: Check if a recipient is on the Peppol network.
- `POST /api/e-invoice/register`: Register a new tenant on the Access Point.
- `POST /internal/sync`: Manual trigger for document synchronization.
